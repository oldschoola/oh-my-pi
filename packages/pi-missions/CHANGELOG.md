# Changelog

All notable changes to `@oh-my-pi/pi-missions` will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

## [Unreleased]

### Fixed (History tab — archived batch visibility)

- **`resolveStatus()` in `dashboard-api.ts` now honors terminal `batch.phase` values even when the top-level `state.completedAt` is unset.** Archived batch missions written to `.omp/missions/<batchId>.json` frequently carry `batch.phase === "complete"`, `"error"`, or `"aborted"` without a sibling `completedAt`. The previous implementation only considered `completedAt`, so those archives were misclassified as `"active"` and never surfaced on the History tab. The summary returned by `/api/missions` now maps `batch.phase === "complete"` → `status: "completed"`, `"aborted"` / `"error"` → `"failed"`, before falling back to the existing `completedAt` / `paused` checks — keeping simple-mission semantics unchanged.

### Tests

- **`src/server.e2e.test.ts` — archived batch history visibility.** Two new tests seed MissionState archives with `batch.phase === "complete"` and `batch.phase === "aborted"` (no `state.completedAt`) and assert `/api/missions` returns `status: "completed"` and `status: "failed"` respectively. Protects the History tab against regressions that hide archived batches behind "active" filtering.

### Added (Start tab — supervisor chat terminal + opus model options)

- **Start tab now renders a supervisor chat terminal below the mission form.** The new terminal polls `/api/supervisor/detail` every 3 s for the shared conversation stream and posts operator messages via the new `POST /api/supervisor/send` endpoint, which appends `{ ts, role: "operator", content }` entries to `.omp/supervisor/conversation.jsonl`. Messages typed here survive into the Active tab's `SupervisorPanel` when a batch is running, giving operators a continuous conversation across tabs. Empty messages (all whitespace) are rejected with `400 empty_message`.
- **`claude-opus-4-7` and `claude-opus-4-6` added to `MODEL_OPTIONS` in `MissionStartForm.tsx`.** Ordered above `claude-opus-4-5` so the newest options surface first in each role dropdown (planner/worker/reviewer).

### Changed (Start tab — defaults)

- **`MissionStartForm` defaults updated.** Template now defaults to `adaptive` (was `standard`); autonomy now defaults to `auto` (was `medium`). Matches operator preference for the "run to completion with phase list growing from plan complexity" flow.

### Tests

- **`src/server.e2e.test.ts` — supervisor send endpoint.** Two new tests assert `POST /api/supervisor/send` appends an `operator` entry that round-trips through `/api/supervisor/detail`'s `conversation` array, and rejects all-whitespace content with `400 empty_message`.

### Changed (Active tab — page-swap layout)

- **Active tab now uses a page-swap layout instead of a 3-column grid.** The mission list is the primary view; clicking a mission swaps the entire view to a full mission detail page with a back button. No mission is auto-selected on initial load or page refresh (`lastSelectedMissionId` preference is no longer restored on bootstrap), so users always see the list first. The inspector rail (Supervisor / Mailbox / Agents) moves to the detail view so list browsing stays focused. Cross-tab navigation from History → Active still opens the mission detail directly via `handleHistorySelect`.

### Fixed (Active batch visibility — V5 PersistedBatchState)

- **`readActiveBatch()` in `dashboard-api.ts` now projects V5 `PersistedBatchState` into the mission list.** Previously it short-circuited and returned `null` whenever `.omp/mission-batch.json` held the engine-side V5 schema (top-level `schemaVersion`/`batchId`, no `description`), so any batch that had reached `executing`/`paused`/`merging` was invisible in `/api/missions` until the MissionState archive was written at completion. The reader now maps `MissionBatchPhase` → wire `BatchPhase` (`executing` → `running`, `stopped` → `aborted`, `completed` → `complete`, `failed` → `error`, `launching` → `planning`) and synthesises a dashboard-shaped `BatchState` with lane/task counts, so active batches show up with correct status badges. `listMissions` de-duplicates when a matching historical archive also exists (active wins), and `loadMissionRecord` prefers the active entry so the dashboard sees live telemetry instead of a stale terminal snapshot.

### Tests

- **`src/server.e2e.test.ts` — V5 active batch visibility.** Two new tests write `.omp/mission-batch.json` in the V5 `PersistedBatchState` shape and assert `/api/missions` surfaces the batch. First test covers `phase: "executing"` → `status: "active"`, `batchPhase: "running"`, plus `tasksTotal`/`tasksComplete` counts. Second test covers `phase: "paused"` → `status: "paused"`, `batchPhase: "paused"`.

### Changed (V2 dashboard — Validation Contract redesign)

- **`ValidationContractView` rewritten as a status-aware panel.** Previously the box grouped assertions by area and listed them flatly; it ignored live milestone status and `updatedAt`. The redesigned panel joins `/validation-contract` + `/validation-status` + `/milestones` client-side so each assertion carries its bound milestone's status (`passed` / `validating` / `in_progress` / `pending` / `failed`) or an `unbound` pill when the assertion is cross-cutting. Header now surfaces `Updated <relative>`, schema version, and assertion count. A proportional status-summary strip + legend shows the shape of the contract at a glance, and a row of area filter pills (with counts) drives a single-area view. Each assertion renders as a card with a status dot, VA-id mono, area chip, copy-id button, 3-line clamp + "Show more" on long descriptions, and a disclosure for acceptance criteria (auto-expanded when ≤ 2). Clicking the milestone chip opens the existing `MilestoneDetailModal` — the same modal `MilestonePanel` uses — populated with the bound milestone and its `boundAssertions` / `findings` from `/validation-status`. Matches the `.milestone-row-clickable` focus-suppression pattern so modal-close refocus doesn't paint a bright ring on the chip.
- **Assertion bindings are computed bidirectionally.** An assertion is bound to a milestone if `assertion.milestoneId === milestone.id` OR `milestone.assertionIds.includes(assertion.id)`. Both edges exist in production data — the contract authors the forward edge, milestones independently list their assertions. The panel scans both directions and takes the first match, matching `validation-contract.ts`'s `assertionsForMilestone` helper.

### Fixed (V2 integration pass — dashboard panels now reflect reality)

- **Telemetry rollup + Agents panel: production workers now flow through `spawnHostedAgent`.** `runTask` in `src/missioncontrol/lane-runner.ts` previously invoked the bare `spawnAgent` adapter, so the adapter's RpcClient events were discarded: every `TaskOutcome.telemetry` stayed `undefined`, and no runtime manifest was ever written for the per-batch registry. The MVP path now goes through `spawnHostedAgent` with a task-scoped `agentId` (`worker-<batchId>-lane-<N>-<taskId>`), writes a manifest + refreshes the registry snapshot per spawn, and attaches a `TaskTelemetry` record (input/output/cache tokens, cost USD, tool calls, duration) to the outcome. The dashboard's `aggregateBatchTelemetry` + `rollupBatchState` + `getMissionAgentStatus` readers now have real data to read instead of the previous zeros.
- **Mailbox panel: `writeMailboxMessage` now emits a `message_sent` audit row.** Every supervisor → agent message (direct and broadcast) drops an entry into `.omp/mailbox/<batchId>/events.jsonl` via `appendMailboxAuditEvent`, carrying `from`, `to`, `messageId`, `messageType`, a 100-char `contentPreview`, and `broadcast` flag. The server's `readMailboxEvents` reader already consumed this file; the writers simply never populated it. `writeBroadcastMessage` inherits the audit through its existing pass-through to `writeMailboxMessage`.
- **Supervisor panel: `createEngine` now activates a supervisor lockfile + heartbeat whenever a batch is running.** New `activateSupervisor(stateRoot, batchId)` / `deactivateSupervisor(handle, stateRoot)` helpers in `src/missioncontrol/supervisor.ts` write `.omp/supervisor/lock.json` with a 30s heartbeat timer (unref'd, non-blocking) and remove it on pause/abort/`onSessionEnd`. `runtime.ts handlers.{batch, resume}` activate; `handlers.{pause, abort}` + `hooks.onSessionEnd` deactivate. The dashboard's `readLockfile` + `isLockStale` path now reflects live state instead of always reporting "inactive".
- **Supervisor timeline: lane-runner hot paths now emit engine lifecycle events.** `startLaneRunner` emits `wave_start` on every wave transition and `batch_complete` when the batch drains; `runTask` emits `task_complete` / `task_failed` after recording each outcome. Events flow through `emitEngineEvent` to `.omp/supervisor/events.jsonl` alongside Tier 0 events. A new `readEngineEventsForBatch` reader in `src/missioncontrol/supervisor.ts` filters engine events by batch, and `getSupervisorDetail` in `src/dashboard-api.ts` merges them into the timeline next to audit entries + Tier 0 events. New `EngineEventSummary` type exported from the engine barrel.
- **`orch_retry_task` counter hygiene.** Now accepts `stalled` in addition to `failed`, decrements `state.failedTasks` (floored at zero), clears `exitDiagnostic` / `partialProgressCommits` / `partialProgressBranch`, and transitions `state.phase` from `failed` to `stopped` so `orch_resume(force=true)` can pick up cleanly.
- **`orch_skip_task` counter hygiene + blocked recomputation.** Now accepts `stalled`, increments `state.skippedTasks`, decrements `failedTasks` when the prior status was `failed`, and drops the skipped task id from `state.blockedTaskIds` (with matching `blockedTasks` count decrement) — previously the counter never moved after a skip.
- **`orch_force_merge` guards.** Added a phase guard that refuses force-merge while the engine still owns the batch (`executing` / `merging` / `planning`); a mixed-outcome guard that rejects force-merge for non-mixed-outcome `failureReason` values (git conflicts, network errors, etc.) where marking the merge `succeeded` would hide real damage; and on success transitions the batch to `paused` + adjusts `skippedTasks` / `failedTasks` counters when `skipFailed=true`.

### Added (Factory alignment — Track #2: engine validator loop integration)
- **Milestone validator driver** (`src/missioncontrol/milestone-runner.ts`). `runValidationRound(state, milestone, deps)` composes the Track B/C validator spawners, the Track F knowledge ingestor, the Track A4 fix-feature generator, and the Track A3 milestone lifecycle reducers into a single end-to-end round. Returns a discriminated `ValidationRoundResult` (`passed | fix-features-generated | failed-max-rounds | failed-fatal | stalled`). Every spawn, parse, persist, generator, and lessons ingestor is injectable for tests. `insertFixFeaturesIntoBatch` and `countFixFeaturesForMilestone` expose the underlying state mutations as pure helpers.
- **Lane-runner hook** (`src/missioncontrol/lane-runner.ts`). When the lane-runner loop finds the current wave drained and the current milestone in `"validating"` status, it invokes `runValidationRound` via `LaneRunnerDeps.validation` (optional override bag for tests) instead of blindly advancing the wave. Passing verdicts mark the milestone passed; `needs-fix` verdicts generate fix features + append a fresh wave + reset the milestone to `"in_progress"` so the existing feature execution path picks them up before the next validator round runs. Stall outcomes pause the loop and log a warning for operator intervention. `LaneRunnerDeps.initialMilestoneReconciliations` accepts a per-milestone reconciliation pool sourced from resume; each entry is consumed exactly once on the lane-runner's first visit to the matching validating milestone.
- **Resume-aware round execution** — `runValidationRound` now accepts an optional `reconciliation: MilestoneValidatorReconciliation` dep. When set (populated by the resume flow, see Track #3), the round number is reused instead of bumped and only validator slots marked `needsRespawn: true` are re-spawned; slots with an existing valid output file on disk skip the spawn and are consumed directly from disk during the parse step. Bridges Track #2 ↔ Track #3.
- **Exports** — `runValidationRound`, `insertFixFeaturesIntoBatch`, `countFixFeaturesForMilestone`, `ValidationRoundDeps`, `ValidationRoundResult`, `MilestoneRuntime` land on the engine barrel.
